# 01 — System Concept

## 1.1 ปัญหาและที่มา
ในสถานการณ์ฉุกเฉินภายในองค์กร (เจ็บป่วยเฉียบพลัน อุบัติเหตุ ไฟไหม้ ภัยคุกคาม
น้ำท่วม ติดอยู่ในอาคาร) บุคลากรมีเวลาและสติจำกัดในการขอความช่วยเหลือ ปัญหาหลักคือ
- โทรหาเบอร์กลางแล้วต้องอธิบายว่าเป็นใคร อยู่ตรงไหน เสียเวลานาที
- บางพื้นที่ในอาคาร/พื้นที่ห่างไกล **ไม่มีสัญญาณอินเทอร์เน็ต**
- ศูนย์ควบคุมไม่มีภาพรวมว่าใครแจ้งเหตุ อยู่ที่ใด สถานะการช่วยเหลือเป็นอย่างไร

## 1.2 วิสัยทัศน์
> “กดปุ่มเดียว องค์กรรู้ทันทีว่า **ใคร–อยู่ที่ไหน–เกิดอะไร** และส่งความช่วยเหลือถึงได้เร็วที่สุด
> ไม่ว่าจะมีอินเทอร์เน็ตหรือไม่”

## 1.3 วัตถุประสงค์ (Objectives)
1. แจ้งเหตุฉุกเฉินได้ภายใน **≤ 5 วินาที** หลังตัดสินใจ (กดค้าง 3 วิ + ส่งอัตโนมัติ)
2. ระบุตัวตนผู้แจ้งและตำแหน่งล่าสุด **อัตโนมัติ** จากบัญชีองค์กร + GPS/Indoor positioning
3. แจ้งเตือนผู้เกี่ยวข้องหลายช่องทางพร้อมกัน (ศูนย์ควบคุม, หัวหน้างาน, รปภ., ผู้ติดต่อฉุกเฉิน)
4. ทำงานได้แม้ **ไม่มีอินเทอร์เน็ต** ผ่าน 6 ช่องทางสำรองตามลำดับความพร้อม
5. ให้ศูนย์ควบคุมมี Dashboard + แผนที่ + Dispatch + Timeline จนปิดเหตุ
6. ปฏิบัติตาม **PDPA** อย่างเคร่งครัด โดยเฉพาะข้อมูลสุขภาพและผู้ติดต่อฉุกเฉิน

## 1.4 ขอบเขต (Scope)
**In scope**
- แอปมือถือสำหรับบุคลากร (Android/iOS) + Web Dashboard ศูนย์ควบคุม
- เชื่อมต่อข้อมูลบุคลากรจาก HRIS/LDAP/AD/Entra ID/Google Workspace (read-only sync)
- ช่องทางสื่อสาร Online + Offline (SMS, BLE Mesh, Wi-Fi Direct, Store&Forward, LoRa, Satellite)
- การแจ้งเตือน: Push, SMS, LINE OA, Email, Dashboard alert
- ระบบความปลอดภัย, RBAC, Audit, Consent, การปฏิบัติตาม PDPA

**Out of scope (เฟสนี้)**
- การเชื่อมต่อตรงกับหน่วยงานภายนอก (1669/191/199) แบบ automated dispatch — เฟสนี้ทำเป็น manual escalation
- ระบบ CCTV/IoT sensor integration (ออกแบบ interface ไว้รองรับ แต่ไม่ implement)
- การวิเคราะห์ AI เชิงคาดการณ์ (predictive) — วางเป็น future roadmap

## 1.5 ผู้มีส่วนได้ส่วนเสีย (Stakeholders) และบทบาท
| บทบาท | ความต้องการหลัก | สิทธิ์โดยสรุป |
|-------|------------------|----------------|
| บุคลากรทั่วไป (Employee) | แจ้งเหตุเร็ว, ปลอดภัย, ข้อมูลส่วนตัวถูกคุ้มครอง | สร้าง SOS ของตน, ดูประวัติของตน, จัดการ consent/ผู้ติดต่อ |
| หัวหน้างาน (Supervisor) | รู้เหตุของลูกทีมทันที, ติดตามสถานะ | ดู/ติดตาม SOS ของผู้ใต้บังคับบัญชา, ประสานงาน |
| เจ้าหน้าที่ รปภ. (Responder) | รับเคส real-time, เห็นตำแหน่งบนแผนที่ | รับเคส, อัปเดตสถานะภาคสนาม, ดูข้อมูลที่จำเป็นต่อการช่วยเหลือ |
| ศูนย์ควบคุม/แอดมิน (Control/Admin) | ภาพรวมทั้งหมด, จัดลำดับ, broadcast, รายงาน | จัดการเคสทั้งหมด, dispatch, ตั้งค่า, จัดการผู้ใช้, ดู audit |
| DPO / Security | กำกับ PDPA, ความปลอดภัย | ดู audit log, จัดการคำขอใช้สิทธิเจ้าของข้อมูล |

## 1.6 หลักการออกแบบ (Design Principles)
- **Reliability over richness** — ความน่าเชื่อถือของการส่งสัญญาณมาก่อนความสวยงาม
- **Offline-first & multi-channel redundancy** — ทุกเส้นทางมี fallback
- **Zero-friction in crisis** — ลด cognitive load: ปุ่มเดียว, ระบุตัวตน/ตำแหน่งอัตโนมัติ
- **Idempotent & deduplicated** — เหตุเดียวกันที่มาหลายช่องทางต้องรวมเป็นเคสเดียว
- **Privacy & security by design/default** — เปิดเผยข้อมูลเท่าที่จำเป็น (least privilege/data minimization)
- **Observable & auditable** — วัดผลได้ ตรวจสอบย้อนหลังได้ทุกการกระทำ

## 1.7 ตัวชี้วัดความสำเร็จ (KPI / SLO)
| ตัวชี้วัด | เป้าหมาย |
|-----------|----------|
| เวลาจากกดปุ่ม → ศูนย์ควบคุมเห็นเคส (online) | P95 ≤ 5 วินาที |
| อัตราการส่งสัญญาณสำเร็จ (อย่างน้อย 1 ช่องทาง) | ≥ 99.9% |
| เวลาตอบสนองเคส (acknowledge) ในเวลาทำการ | P90 ≤ 60 วินาที |
| เวลาตั้งแต่ offline กลับมา online → sync เข้าเคส | P95 ≤ 30 วินาที |
| False trigger rate (กดผิด/ยกเลิกทัน) | ≤ 2% และมี cancel window |
| ความพร้อมใช้งานระบบ (availability) | ≥ 99.9%/เดือน |
| การละเมิดข้อมูลส่วนบุคคล (PDPA breach) | 0 |

## 1.8 สมมติฐานและข้อจำกัด (Assumptions & Constraints)
- บุคลากรมีสมาร์ทโฟนที่รองรับ BLE/Wi-Fi Direct และติดตั้งแอปไว้ล่วงหน้า
- องค์กรมีระบบ Identity Provider (IdP) ที่รองรับ OIDC/SAML หรือ LDAP/AD
- ช่องทาง LoRa/Satellite ใช้ได้เฉพาะเมื่อองค์กรลงทุนอุปกรณ์เสริม (optional)
- iOS มีข้อจำกัดเรื่อง background BLE/Wi-Fi Direct มากกว่า Android → ออกแบบ degrade gracefully
- การส่ง SMS ขึ้นกับความครอบคลุมเครือข่ายมือถือในพื้นที่
