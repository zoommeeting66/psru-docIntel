# 10 — PDPA Compliance

อ้างอิง พ.ร.บ.คุ้มครองข้อมูลส่วนบุคคล พ.ศ. 2562 (PDPA ของไทย)
หลักการ: **Lawfulness · Purpose limitation · Data minimization · Privacy by design/default**

> หมายเหตุ: เอกสารนี้เป็นแนวออกแบบเชิงเทคนิค/กระบวนการ ไม่ใช่คำปรึกษากฎหมาย
> ควรให้ DPO/ฝ่ายกฎหมายขององค์กรตรวจรับรองก่อน go-live

## 10.1 บทบาทตามกฎหมาย
- **ผู้ควบคุมข้อมูล (Data Controller)**: องค์กรเจ้าของระบบ
- **ผู้ประมวลผลข้อมูล (Data Processor)**: ผู้ให้บริการคลาวด์/ผู้พัฒนา (มี DPA สัญญา)
- **DPO**: แต่งตั้งเจ้าหน้าที่คุ้มครองข้อมูล กำกับการปฏิบัติ

## 10.2 ฐานการประมวลผล (Legal Basis)
| ข้อมูล/กิจกรรม | ฐานทางกฎหมาย |
|----------------|----------------|
| ข้อมูลบุคลากรพื้นฐาน (ชื่อ, หน่วยงาน, เบอร์) จาก HR | สัญญาจ้าง/ภารกิจองค์กร (legitimate interest) |
| ตำแหน่งขณะแจ้งเหตุ | **ความยินยอม** + ประโยชน์สำคัญต่อชีวิต (vital interest) |
| ข้อมูลสุขภาพ (sensitive) | **ความยินยอมโดยชัดแจ้ง** + vital interest ขณะฉุกเฉิน |
| ผู้ติดต่อฉุกเฉิน (ข้อมูลบุคคลที่สาม) | ความยินยอม + แจ้งผู้ติดต่อตามสมควร |
| Audit log | หน้าที่ตามกฎหมาย/ประโยชน์โดยชอบ (security) |

> ข้อมูลสุขภาพเป็น **ข้อมูลอ่อนไหว (มาตรา 26)** ต้องขอความยินยอมโดยชัดแจ้ง
> และจำกัดการเข้าถึงเฉพาะผู้เกี่ยวข้องในภาวะฉุกเฉิน

## 10.3 ความยินยอม (Consent Management)
- แยก consent เป็นราย **purpose**: `emergency_health_disclosure`, `emergency_contact`,
  `location_tracking` — ผู้ใช้เลือกได้อิสระ (granular, ไม่ bundling)
- บันทึก: เวอร์ชันข้อความ consent, เวลาที่ให้/ถอน, IP, scope (เปิดเผยอะไรบ้าง)
- **ถอนได้ทุกเมื่อ** ในแอป (`DELETE /v1/me/consents/:id`) → หยุดประมวลผล + ปิดบังข้อมูลทันที
- เก็บหลักฐานการยินยอม (ตาราง `consent`) เพื่อพิสูจน์ accountability

## 10.4 Data Minimization & Need-to-know
- เก็บเฉพาะข้อมูลที่จำเป็นต่อการช่วยเหลือ
- ข้อมูลสุขภาพ/ผู้ติดต่อ **ปกติแสดงแบบ masked** บน dashboard;
  responder/control เปิดดูได้เฉพาะเมื่อมีเคส active + ระบุ purpose → **บันทึก audit ทุกครั้ง**
- รายงานทั่วไป/สถิติ ใช้ข้อมูล **anonymized/pseudonymized** (ตัดชื่อ, รวมเป็นกลุ่ม)

## 10.5 สิทธิเจ้าของข้อมูล (Data Subject Rights — DSAR)
| สิทธิ | การรองรับในระบบ |
|------|------------------|
| เข้าถึง/ขอสำเนา (access) | `GET /v1/me/*`, ปุ่ม "ดาวน์โหลดข้อมูลของฉัน" |
| แก้ไขให้ถูกต้อง (rectification) | แก้ profile/health/contacts ในแอป (ข้อมูลจาก HR แก้ที่ต้นทาง) |
| ลบ/ถูกลืม (erasure) | flow DSAR → soft delete + anonymize ตาม retention; คงไว้เฉพาะที่กฎหมายบังคับ |
| ระงับการใช้ (restriction) | ถอน consent → ปิดบัง/หยุดประมวลผล |
| คัดค้าน (objection) | ตั้งค่าปิดการแชร์ที่ไม่ใช่ vital interest |
| โอนย้าย (portability) | export JSON/CSV ข้อมูลของตน |
- ช่องทาง DSAR: ในแอป + อีเมล DPO; SLA ตอบสนองตามที่กฎหมาย/นโยบายกำหนด (เช่น ภายใน 30 วัน)

## 10.6 Retention & Disposal
| ข้อมูล | ระยะเก็บ (ตัวอย่าง — องค์กรกำหนดจริง) | เมื่อครบ |
|--------|------|----------|
| Incident operational | 2 ปี | anonymize (คงสถิติ ไม่ระบุตัวตน) |
| ข้อมูลสุขภาพ | จนกว่าถอน consent / พ้นสภาพ + grace | ลบ/ปิดบัง |
| ผู้ติดต่อฉุกเฉิน | จนกว่าถอน / พ้นสภาพ | ลบ |
| Audit log | 1–2 ปี (ตามนโยบาย/กฎหมาย) | จัดเก็บ cold แล้วลบ |
| Outbox (mobile) | จนส่งสำเร็จ + 7 วัน | ลบจากเครื่อง |
- ลบจริงแบบ secure (crypto-shredding: ทำลาย key = ทำลายข้อมูลที่เข้ารหัส)

## 10.7 ROPA (บันทึกรายการกิจกรรมประมวลผล)
จัดทำ Record of Processing Activities ครอบคลุม: วัตถุประสงค์, ประเภทข้อมูล,
กลุ่มเจ้าของข้อมูล, ผู้รับข้อมูล, การส่งข้ามแดน, ระยะเก็บ, มาตรการความปลอดภัย
(ทบทวนเมื่อมีฟีเจอร์/integration ใหม่)

## 10.8 การส่งข้อมูลข้ามแดน (Cross-border)
- เลือก hosting ที่ **data residency** ในไทย/ภูมิภาคที่เหมาะสม
- หากใช้ผู้ให้บริการต่างประเทศ (FCM/APNs/LINE/SMS) → ส่งเฉพาะข้อมูลน้อยที่สุด
  (push token, ข้อความแจ้งเตือนแบบไม่ระบุรายละเอียดอ่อนไหว) + มี DPA/มาตรฐานคุ้มครอง

## 10.9 Notification เนื้อหาปลอดภัย
- Push/SMS/LINE/Email **ไม่ใส่ข้อมูลสุขภาพ/รายละเอียดอ่อนไหว** ในตัวข้อความ
  ("มีเหตุฉุกเฉิน โปรดเปิดแอป" + ลิงก์เข้าระบบที่ authenticate) — กันข้อมูลรั่วทาง notification

## 10.10 DPIA (Data Protection Impact Assessment)
ระบบเข้าข่ายต้องทำ DPIA (ข้อมูลสุขภาพ + ติดตามตำแหน่ง):
- ประเมินความเสี่ยงต่อสิทธิเสรีภาพ + มาตรการลด (encryption, need-to-know, audit, consent)
- ทบทวน DPIA เมื่อเพิ่มช่องทาง/ฟีเจอร์ที่กระทบความเป็นส่วนตัว

## 10.11 Breach Management
- กระบวนการตรวจจับ (SIEM) → ประเมิน → แจ้งคณะกรรมการคุ้มครองข้อมูลฯ **ภายใน 72 ชม.**
  (ถ้าเข้าเกณฑ์) + แจ้งเจ้าของข้อมูลหากเสี่ยงสูง
- Runbook + ผู้รับผิดชอบ + log หลักฐาน (เชื่อมกับ Security §9 และ Maintenance §15)

## 10.12 Privacy by Design/Default — สรุปที่ฝังในระบบ
- Schema แยกข้อมูลอ่อนไหว + เข้ารหัส field-level (default)
- Consent ต้องมีก่อนเก็บ/แสดงข้อมูลอ่อนไหว
- Masking เป็นค่าเริ่มต้น, เปิดดู = audit
- รายงานทั่วไป anonymized โดยปริยาย
- ทุกการเข้าถึงข้อมูลส่วนบุคคลถูกบันทึก
